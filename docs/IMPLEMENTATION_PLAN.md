# SFDT Implementation Plan

## Vision

SFDT (Salesforce DevTools) is a production-grade CLI for Salesforce DX projects that packages deployment, testing, quality analysis, and release management into a single installable tool. Instead of maintaining dozens of scattered shell scripts across projects, teams install `@sfdt/cli` once and get a consistent, configurable interface for every DevOps workflow: deploying to orgs, running tests with failure analysis, detecting metadata drift, generating release manifests, and more. AI-powered features (code review, test failure analysis, release notes) are available when the Claude CLI is present and degrade gracefully when it is not.

## Architecture

SFDT uses a hybrid architecture: a Node.js CLI layer (Commander.js) handles argument parsing, configuration loading, and user interaction, while battle-tested shell scripts perform the actual Salesforce CLI operations. This design preserves existing script investments while adding structured configuration, consistent output formatting, and optional AI integration on top.

```
┌─────────────────────────────────────────────────────────┐
│                        User                             │
│                    sfdt <command>                        │
└───────────────────────┬─────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────────┐
│                   bin/sfdt.js                            │
│              (Node.js entry point)                       │
└───────────────────────┬─────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────────┐
│                   src/cli.js                             │
│              Commander.js program                        │
│         Registers all command modules                    │
└──────┬──────────┬───────────┬──────────┬────────────────┘
       │          │           │          │
       ▼          ▼           ▼          ▼
┌──────────┐ ┌────────┐ ┌────────┐ ┌─────────┐
│ commands/│ │  lib/  │ │  lib/  │ │  lib/   │
│ deploy.js│ │config  │ │output  │ │  ai.js  │
│ test.js  │ │  .js   │ │  .js   │ │(Claude) │
│ etc.     │ │        │ │        │ │         │
└────┬─────┘ └───┬────┘ └────────┘ └─────────┘
     │           │
     ▼           ▼
┌─────────────────────────────────────────────────────────┐
│               src/lib/script-runner.js                   │
│                                                         │
│  1. Loads .sfdt/config.json + related config files      │
│  2. Flattens config into SFDT_* environment variables   │
│  3. Spawns shell script via execa with:                 │
│     - TTY passthrough (stdio: 'inherit')                │
│     - Merged env (process.env + SFDT_* vars)            │
│     - Working directory set to project root             │
└───────────────────────┬─────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────────┐
│                  scripts/                                │
│                                                         │
│  core/deployment-assistant.sh                           │
│  core/enhanced-test-runner.sh                           │
│  core/generate-release-manifest.sh                      │
│  quality/code-analyzer.sh                               │
│  new/preflight.sh                                       │
│  ...etc                                                 │
│                                                         │
│  Reads SFDT_* env vars for all configuration.           │
│  Uses sf CLI, git, and standard UNIX tools.             │
└─────────────────────────────────────────────────────────┘
```

Key architectural decisions:

- **execa** spawns scripts with `stdio: 'inherit'` for full TTY passthrough, preserving interactive prompts and colored output from shell scripts.
- **Configuration** lives in `.sfdt/` per project, generated by `sfdt init`. Multiple JSON files are merged into a single config object at runtime.
- **SFDT_ environment variables** are the contract between Node.js and shell scripts. `script-runner.js` flattens the config object into `SFDT_PROJECT_ROOT`, `SFDT_DEFAULT_ORG`, `SFDT_TEST_COVERAGE_THRESHOLD`, etc.
- **AI features** are optional. `ai.js` checks for the Claude CLI at runtime and returns `null` when unavailable, allowing every AI-enhanced command to proceed without it.
- **No transpilation** required. The package is pure ESM (`"type": "module"` in package.json), targeting Node.js 18+.

## Config Schema

The `.sfdt/` directory contains four configuration files, generated by `sfdt init` and merged at runtime by `config.js`.

### `.sfdt/config.json` (primary)

```json
{
  "projectName": "my-salesforce-project",
  "defaultOrg": "my-dev-org",
  "features": {
    "ai": true,
    "notifications": false,
    "releaseManagement": true
  },
  "deployment": {
    "keyClasses": ["AccountTriggerHandler", "OpportunityService"],
    "coverageThreshold": 75,
    "excludedProfiles": ["Admin"]
  },
  "ai": {
    "provider": "claude",
    "allowedTools": ["Bash(git*)", "Bash(gh*)"]
  },
  "notifications": {
    "slack": {
      "webhookUrl": "https://hooks.slack.com/services/T.../B.../..."
    }
  },
  "smokeTests": {
    "testClasses": ["SmokeTest_AccountCreate", "SmokeTest_FlowValidation"],
    "healthChecks": ["/services/data/vXX.0/limits"]
  }
}
```

| Field | Type | Description |
|-------|------|-------------|
| `projectName` | string | Display name for the project, used in notifications and output |
| `defaultOrg` | string | Default Salesforce org alias for all commands |
| `features.ai` | boolean | Enable/disable AI-powered features (code review, test analysis, release notes) |
| `features.notifications` | boolean | Enable/disable Slack notification integration |
| `features.releaseManagement` | boolean | Enable/disable release manifest generation features |
| `deployment.keyClasses` | string[] | Critical Apex classes to validate during deployment |
| `deployment.coverageThreshold` | number | Minimum code coverage percentage required (0-100) |
| `deployment.excludedProfiles` | string[] | Profile names to exclude from deployment manifests |
| `ai.provider` | string | AI provider identifier (currently only `"claude"`) |
| `ai.allowedTools` | string[] | Tools the AI agent is permitted to use during prompts |
| `notifications.slack.webhookUrl` | string | Slack Incoming Webhook URL for notifications |
| `smokeTests.testClasses` | string[] | Apex test classes to run during smoke testing |
| `smokeTests.healthChecks` | string[] | REST API endpoints to verify during smoke testing |

### `.sfdt/environments.json`

```json
{
  "default": "my-dev-org",
  "orgs": [
    {
      "alias": "my-dev-org",
      "type": "development",
      "description": "Default development org",
      "timeout": 300,
      "parallelTests": true,
      "coverageThreshold": 75
    },
    {
      "alias": "my-staging-org",
      "type": "staging",
      "timeout": 600,
      "parallelTests": true,
      "coverageThreshold": 75
    },
    {
      "alias": "my-prod-org",
      "type": "production",
      "timeout": 900,
      "parallelTests": false,
      "coverageThreshold": 90
    }
  ]
}
```

| Field | Type | Description |
|-------|------|-------------|
| `default` | string | Alias of the default org (matches an entry in `orgs`) |
| `orgs[].alias` | string | Salesforce CLI org alias |
| `orgs[].type` | string | Environment type: `development`, `staging`, or `production` |
| `orgs[].description` | string | Human-readable description of the org's purpose |
| `orgs[].timeout` | number | Command timeout in seconds for this environment |
| `orgs[].parallelTests` | boolean | Whether to run tests in parallel on this org |
| `orgs[].coverageThreshold` | number | Override code coverage threshold for this org |

### `.sfdt/pull-config.json`

```json
{
  "metadataTypes": [
    "ApexClass", "ApexTrigger", "LightningComponentBundle",
    "CustomObject", "CustomField", "Layout",
```

| Field | Type | Description |
|-------|------|-------------|
| `metadataTypes` | string[] | Metadata types to include in pull operations |

| `priorityMetadataTypes` | string[] | Metadata types to pull first (higher priority) |
| `excludedMetadataTypes` | string[] | Metadata types to always exclude from pulls |
| `excludedProfiles` | string[] | Profile names to exclude from pull operations |
| `pullGroups` | object | Named groups of metadata types for selective pulling |

### `.sfdt/test-config.json`

```json
{
  "coverageThreshold": 75,
  "testLevel": "RunLocalTests",
```

## Environment Variables

`script-runner.js` flattens the merged config object into `SFDT_*` environment variables before spawning each shell script. Scripts read these variables instead of parsing JSON directly.

| Variable | Source | Description |
|----------|--------|-------------|
| `SFDT_PROJECT_ROOT` | `config._projectRoot` | Absolute path to the Salesforce DX project root |
| `SFDT_CONFIG_DIR` | `config._configDir` | Absolute path to the `.sfdt/` configuration directory |
| `SFDT_DEFAULT_ORG` | `config.defaultOrg` | Default Salesforce org alias |
| `SFDT_DEFAULT_ENV` | `config.environments.default` | Default environment name from environments config |
| `SFDT_ENV_ORGS` | `config.environments.orgs` | Comma-separated list of org aliases from all environments |
| `SFDT_FEATURE_AI` | `config.features.ai` | Whether AI features are enabled (`true`/`false`) |
| `SFDT_FEATURE_NOTIFICATIONS` | `config.features.notifications` | Whether Slack notifications are enabled (`true`/`false`) |
| `SFDT_FEATURE_RELEASE_MANAGEMENT` | `config.features.releaseManagement` | Whether release management is enabled (`true`/`false`) |
| `SFDT_TEST_COVERAGE_THRESHOLD` | `config.testConfig.coverageThreshold` | Minimum code coverage percentage |
| `SFDT_TEST_LEVEL` | `config.testConfig.testLevel` | Salesforce test level (`RunLocalTests`, etc.) |
| `SFDT_TEST_SUITES` | `config.testConfig.suites` | Comma-separated list of test suite names |
| `SFDT_PULL_METADATA_TYPES` | `config.pullConfig.metadataTypes` | Comma-separated list of metadata types to pull |
| `SFDT_PULL_TARGET_DIR` | `config.pullConfig.targetDir` | Target directory for pulled metadata |
| `SFDT_TARGET_ORG` | (per-command) | Target org alias, set by `--org` flag on commands like `smoke`, `rollback`, `drift` |
| `SFDT_PREFLIGHT_STRICT` | (per-command) | When `true`, preflight treats warnings as errors |

## Risk Register

| Risk | Severity | Mitigation |
|------|----------|------------|
| Shell scripts behave differently across macOS and Linux | High | Test on both platforms in CI. Use POSIX-compatible constructs. Document bash 3.2+ requirement for macOS. |
| `sf` CLI version incompatibilities | High | Detect `sf` version at runtime in `preflight`. Document minimum supported version. Test against sf v2 LTS. |
| Claude CLI not installed or not authenticated | Medium | `ai.js` caches availability check and returns `null`. Every AI integration point has a non-AI fallback path. |
| Shell script TTY passthrough breaks in CI environments | Medium | Default `interactive: true` for local use. Scripts detect `$TERM` and degrade to non-interactive output. CI pipelines set `SFDT_NON_INTERACTIVE=true`. |
| Config schema changes between versions | Medium | Validate config on load with descriptive errors. `sfdt init` can regenerate config. Add migration logic when schema changes. |
| Large monorepo projects with multiple package directories | Medium | `project-detect.js` already supports multiple `packageDirectories`. Test with multi-package projects in Phase 5. |
| npm publish with missing files or broken bin entry | Medium | `"files"` whitelist in `package.json`. `prepublishOnly` runs tests and lint. Test with `npm pack --dry-run` before publish. |
| Slack webhook URL leaked in config | Low | Document adding `.sfdt/*.local.json` to `.gitignore`. Consider supporting `SFDT_SLACK_WEBHOOK_URL` env var as alternative to storing URL in config. |
| execa major version breaking changes | Low | Pin execa to `^9.x`. Test in CI. execa 9 is stable ESM. |
| Script execution permissions on fresh clone | Low | `script-runner.js` calls `chmod 755` on every script before execution. |

## Future Roadmap (post-1.0)

These are potential directions after the initial stable release. None are committed.

- **sf CLI plugin integration** — Package sfdt as a native `sf` plugin so it can be installed via `sf plugins install @sfdt/cli` and invoked as `sf sfdt deploy`. This would provide tighter integration with the Salesforce CLI ecosystem.

- **Windows support** — Port shell scripts to Node.js equivalents or use cross-platform shell via `execa`. Currently sfdt requires bash, limiting it to macOS and Linux (or WSL on Windows).

- **Plugin system for community extensions** — Allow third-party packages to register additional sfdt commands. A plugin would export a `register(program)` function that receives the Commander.js instance.

- **Dashboard and report generation** — Generate HTML reports for test history, deployment history, and code quality trends. Store historical data in `.sfdt/history/` and render with a lightweight template engine.

- **Config drift alerting** — Scheduled drift detection (via cron or CI) that compares local source to org state and sends notifications when untracked changes appear. Build on the existing `sfdt drift` command.

- **Multi-package directory support** — Enhanced handling for Salesforce projects with multiple package directories, including selective deployment and per-package test execution.

- **CI/CD pipeline templates** — Provide ready-to-use GitHub Actions workflows and GitLab CI configurations that use sfdt commands. `sfdt init --ci` could scaffold these files.

- **Org comparison** — Compare metadata between two orgs (e.g., staging vs production) to identify configuration differences before promotion.

- **Deployment history tracking** — Record every deployment (timestamp, org, components, result) in `.sfdt/history/deployments.json` for audit and rollback reference.

- **Interactive TUI mode** — A terminal UI (using ink or blessed) for navigating commands, viewing test results, and monitoring deployments in real time.
