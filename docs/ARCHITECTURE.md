# SFDT Architecture & Developer Guide

> For end-user command walkthroughs and workflow examples, see [USAGE.md](USAGE.md).

## Vision

SFDT (Salesforce DevTools) is a production-grade CLI for Salesforce DX projects that packages deployment, testing, quality analysis, and release management into a single installable tool. Instead of maintaining dozens of scattered shell scripts across projects, teams install `@sfdt/cli` once and get a consistent, configurable interface for every DevOps workflow: deploying to orgs, running tests with failure analysis, detecting metadata drift, generating release manifests, and more. AI-powered features (code review, test failure analysis, release notes) are available when the Claude CLI is present and degrade gracefully when it is not.

## Architecture

SFDT uses a hybrid architecture: a Node.js CLI layer (Commander.js) handles argument parsing, configuration loading, and user interaction, while battle-tested shell scripts perform the actual Salesforce CLI operations. This design preserves existing script investments while adding structured configuration, consistent output formatting, and optional AI integration on top.

### Key Technical Decisions

- **execa** spawns scripts with `stdio: 'inherit'` for full TTY passthrough, preserving interactive prompts and colored output from shell scripts.
- **Configuration** lives in `.sfdt/` per project, generated by `sfdt init`. Multiple JSON files are merged into a single config object at runtime.
- **SFDT_ environment variables** are the contract between Node.js and shell scripts. `script-runner.js` flattens the config object into `SFDT_PROJECT_ROOT`, `SFDT_DEFAULT_ORG`, `SFDT_TEST_COVERAGE_THRESHOLD`, etc.
- **AI features** are optional. `ai.js` checks for the Claude CLI at runtime and returns `null` when unavailable, allowing every AI-enhanced command to proceed without it.
- **Org inventory** (`org-inventory.js`) fetches a metadata member inventory from a Salesforce org via `sf org list metadata` or from local source files via glob. Results are returned as `Map<type, Set<member>>`.
- **Org diff** (`org-diff.js`) compares two inventory maps and returns a flat array of `{ type, member, status }` objects where `status` is `source-only`, `target-only`, or `both`.
- **Command runner (SSE)** — the GUI server exposes `GET /api/command/run?command=<name>` which streams shell script output as Server-Sent Events. This lets the dashboard run preflight, drift, and smoke scripts without a separate terminal session.
- **No transpilation** required. The package is pure ESM (`"type": "module"` in package.json), targeting Node.js 18+.

## Package Structure

```
@sfdt/cli
├── bin/
│   └── sfdt.js                          # #!/usr/bin/env node entry point
├── src/
│   ├── cli.js                           # Commander.js program, registers all commands
│   ├── commands/                        # Command-specific logic and UI (Commander.js actions)
│   ├── lib/                             # Shared Node.js libraries (config, AI, etc.)
│   └── templates/                       # Default config files for 'sfdt init'
├── scripts/                             # Core Salesforce logic (Bash/SFDX)
│   ├── core/                            # Deployment, testing, and release logic
│   ├── lib/                             # Shared shell utilities
│   ├── new/                             # Preflight, drift, and smoke test logic
│   ├── quality/                         # Static analysis and test quality logic
│   └── utils/                           # Common shell utility functions
├── test/                                # Vitest test suite
├── package.json                         # Build, dependencies, and npm scripts
└── README.md                            # End-user documentation
```

## Developer Guide

### Environment Setup

```bash
git clone https://github.com/scoobydrew83/sfdt.git
cd sfdt
npm install
npm link
```

After `npm link`, you can run `sfdt` from anywhere on your machine to test local changes.

### Adding a New Command

1.  Create the command action in `src/commands/<name>.js`.
2.  Register the command in `src/cli.js`.
3.  (Optional) Create the supporting shell script in `scripts/`.
4.  Add a unit test in `test/commands/<name>.test.js`.

### Running Tests

```bash
npm test              # Run all tests (vitest)
npm run test:coverage # Generate coverage report
npm run lint          # Run ESLint
```

### CI/CD and Publishing

SFDT uses **GitHub Actions** for CI and **npm Trusted Publishers** for automated releases.
- **CI**: Runs on every PR to `main` (linting, audit, tests).
- **Publish**: Triggered automatically when a version tag (`v*.*.*`) is pushed to `main`.
- **Environment**: Published to npm as `@sfdt/cli`.

## Command Reference

| Command | Script(s) | Description | Key Options |
|---------|-----------|-------------|-------------|
| `sfdt init` | (built-in) | Initialize `.sfdt/` configuration for a Salesforce DX project | |
| `sfdt deploy` | `core/deployment-assistant.sh`, `core/deploy-manager.sh` | Deploy metadata to a Salesforce org | `--managed` |
| `sfdt test` | `core/enhanced-test-runner.sh`, `core/run-tests.sh` | Run Apex tests with optional AI failure analysis | `--legacy`, `--analyze` |
| `sfdt pull` | (built-in, Node.js) | Pull metadata changes from the default org with SQLite delta cache and parallel retrieval | `--full`, `--dry-run`, `--status` |
| `sfdt release` | `core/generate-release-manifest.sh` | Generate a release manifest and optional AI release notes | `[version]` |
| `sfdt changelog` | (built-in) | Manage CHANGELOG.md with AI generation and release automation | `generate`, `release`, `check` |
| `sfdt quality` | `quality/code-analyzer.sh`, `quality/test-analyzer.sh` | Run code quality analysis with optional AI fix plan | `--tests`, `--all`, `--fix-plan` |
| `sfdt preflight` | `new/preflight.sh` | Run pre-deployment validation checks | `--strict` |
| `sfdt rollback` | `new/rollback.sh` | Roll back a deployment to a target org | `--org <alias>` |
| `sfdt smoke` | `new/smoke.sh` | Run post-deployment smoke tests | `--org <alias>` |
| `sfdt drift` | `new/drift.sh` | Detect metadata drift between local source and a target org | `--org <alias>` |
| `sfdt compare` | (built-in, uses `org-inventory.js` + `org-diff.js`) | Compare metadata between two orgs or local source vs an org; optionally writes a `package.xml` of source-only items | `--source <alias\|local>`, `--target <alias>`, `--output <file>` |
| `sfdt review` | (AI-only) | AI-powered Salesforce code review of current branch changes | `--base <branch>` |
| `sfdt notify` | (built-in) | Send a Slack notification for deployment events | `<event>`, `--version`, `--org`, `--message` |

## Config Schema

The `.sfdt/` directory contains four configuration files, generated by `sfdt init` and merged at runtime by `config.js`.

### `.sfdt/config.json` (primary)

Controls global project behavior, target orgs, and feature toggles (AI, Slack, etc.).

| Field | Type | Description |
|-------|------|-------------|
| `projectName` | string | Display name for the project |
| `defaultOrg` | string | Default Salesforce org alias |
| `features.ai` | boolean | Enable/disable AI-powered features |
| `features.notifications` | boolean | Enable/disable Slack notification integration |
| `notifications.slack.webhookUrl` | string | Slack Incoming Webhook URL |

### `.sfdt/environments.json`

Defines named environments (development, staging, production) with environment-specific settings like timeouts and coverage overrides.

### `.sfdt/pull-config.json`

Configures metadata pull groups and excluded metadata types for selective org-to-local synchronization.

### `.sfdt/test-config.json`

Specifies Apex test suites, test levels, and auto-detected test/Apex classes for efficient test execution.
