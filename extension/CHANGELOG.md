## [Unreleased]

### Added
- **Debug Logs — Analyze view: profiler timings + limits + inventories (P3-3).** The Debug Logs viewer gains a per-log **📊 Analyze** action (next to the row's show-body click) that fetches the log body (same worker-proxied path — no second fetch route), runs the P3-2 `parseApexLog`, and presents a profiler overlay. It shows a **method-timing table** (Method / Namespace / Total / Self / Count) aggregated per unique method/code-unit — durations in ms with the raw ns on hover — that's **sortable** by Total, Self, or Count via keyboard-activatable header buttons (with `aria-sort`); **per-namespace governor-limit snapshots** rendered metric → `used / max`; and **SOQL / DML / callout inventories**, each entry showing its payload plus a **"line N" jump control** that scrolls to and highlights that line in the raw log body shown alongside. A token-styled **truncation banner** (`role="alert"`) names the reason when the log was truncated. The overlay is keyboard-accessible (Esc closes, focus restored to the opener, native controls), correct in both themes via `var(--sfdt-color-*)`, and built with `createElement`/`textContent` only (zero `innerHTML`). A new pure `lib/apex-log/viewmodel.ts` maps a `ParsedLog` → analyzer viewmodel (method aggregation + sort helper) with zero DOM and no re-parsing. Extends the existing `debug-log-viewer` feature — **no new feature, permission, or runtime dependency**.
- **Apex log parser — benchmarked big-log strategy (P3-2, PR-3 of 3, closes P3-2).** Internal groundwork, no user-facing UI yet. Settles board AC-3 ("parses a 5 MB log without blocking the UI thread — strategy documented") by **measurement, not assumption**: a new runtime benchmark (`test/apex-log-parser.bench.test.ts`) generates a realistic ~5.3 MB log (no committed fixture) and asserts the pure synchronous `parseApexLog` parses it well under a generous 2000 ms CI budget — **measured ~33 ms on dev hardware** (~60× headroom). Because a 33 ms parse doesn't block the UI thread, the documented strategy is **run it synchronously — no Web Worker**: a worker would have to structured-clone the whole `ParsedLog` back over `postMessage` (O(result size), potentially costing as much as the parse it offloaded), and a chunked-yield parse would force a resumable state machine — neither is justified by the measurement. The benchmark also proves `collectEvents:false` is a real memory lever (drops the flat `events[]` while the tree + inventories stay identical). Rationale, the measured time, and how a consumer (P3-3) should invoke it for large logs (including hosting the *unchanged* pure function in a Worker for pathological logs) are documented in `lib/apex-log/README.md`. `parseApexLog` is unchanged; no new permission, no new runtime dependency.
- **Apex log parser — governor-limit snapshots + SOQL/DML/callout inventories (P3-2, PR-2 of 3).** Internal groundwork, no user-facing UI yet. `parseApexLog` now populates the four inventories PR-1 left empty, still in the same single pure/synchronous pass (zero DOM/IO/`chrome.*`, no new dependency). It collects a **SOQL/DML/callout inventory** — each `SOQL_EXECUTE_BEGIN`/`SOQL_EXECUTE_END`, `DML_BEGIN`/`DML_END`, and `CALLOUT_REQUEST`/`CALLOUT_RESPONSE` pair becomes an entry with its 0-based raw-log line (matching `events`/nodes, for deep-linking), nanosecond timestamp, a back-reference to the enclosing invocation node, and its payload (query + rows; op + sobject + rows; method + endpoint + status). A missing END (truncation) keeps the entry with its END-derived field null. It also captures **per-namespace governor-limit snapshots** from `LIMIT_USAGE_FOR_NS` blocks — reading the indented `label: used out of max` body lines into stable camelCase metric keys (`soqlQueries`, `queryRows`, `dmlStatements`, `cpuTime`, `heapSize`, …; unmapped labels kept under a normalized fallback key so nothing is dropped), flagged `cumulative` when inside a `CUMULATIVE_LIMIT_USAGE` wrapper and kept in document order. Field extraction is defensive across API-version shape variation. The big-log worker/chunking strategy remains PR-3. No new permission, no new runtime dependency.
- **Apex log parser core (P3-2, PR-1 of 3).** Internal groundwork, no user-facing UI yet. A pure, synchronous, dependency-free Apex debug-log parser in `lib/apex-log/` (`parseApexLog(raw) → ParsedLog`) — zero DOM, zero IO, zero `chrome.*`. Single pass over the log lines: parses the header (`apiVersion` + `category→level` debug-level map), captures every event into a flat `events[]` (clock time + nanosecond timestamp + fields), and builds an invocation tree from ENTRY/EXIT tokens (`EXECUTION_STARTED`/`CODE_UNIT_STARTED`/`METHOD_ENTRY`) via an explicit stack, computing per-node `totalNanos` and `selfNanos = total − Σ children` at pop time. Degrades gracefully on all three truncation shapes (`SKIPPED_BYTES`, `MAXIMUM DEBUG LOG SIZE REACHED`, abrupt EOF) — flags `truncated`/`truncationReason`, synthetically closes dangling frames with null durations, and never throws. The `limits`/`soql`/`dml`/`callout` inventories are typed but return empty (PR-2); the big-log worker/chunking strategy is PR-3. No new permission, no new runtime dependency.
- **TraceFlag & debug-session manager (P3-1).** A new **Trace Flags** tool — a Workspace tool and a **"⚑ Trace flags"** header entry in the Debug Logs viewer — for managing debug tracing without a trip to Setup. It **lists trace flags** (querying `TraceFlag` for `TracedEntityId`/`DebugLevelId`/`LogType`/`StartDate`/`ExpirationDate`, resolving the entity id → user name and the level id → DebugLevel name) and shows a **live per-flag expiry countdown** that ticks every second; expired flags are styled distinct (error token + dimmed) and are one-click **renewable** (PATCHes the flag's window forward, respecting the 24h DEVELOPER_LOG cap). You can **start or stop a debug session for yourself or another user** — a native user search (by Name/Username, active users only) plus a **Me** button — where starting creates a `TraceFlag` (and, on demand, a feature-managed `DebugLevel`) and stopping deletes it. Three **DebugLevel presets** — **Basic**, **Full** (every category FINEST), and **Custom** (per-category `ApexCode`/`ApexProfiling`/`Callout`/`Database`/`System`/`Validation`/`Visualforce`/`Workflow` selects) — with the Custom map persisted to `chrome.storage.local` and reapplied exactly on the next session. All Tooling reads/writes go through the worker-proxied `api` client, so the `sid` never leaves the worker; the countdown interval is cleared on close/teardown so no orphan timer survives a route change. Native labelled controls throughout (full keyboard path, both themes via `var(--sfdt-color-*)` tokens). **No new permission** (Tooling API rides on the existing session/`host_permissions`), no new runtime dependency.
- **Docked side panel — org-follow, auto-enable, and a default-surface setting (P2-3, PR-2 of 2).** The docked panel now **follows the active org across tab switches**: with the panel open, moving to a different Salesforce org's tab re-targets the panel to that org and re-renders its tools, in place (no page navigation — the panel document stays alive, so no flash; open tool tabs reset on the org change). Switching between two hostnames of the *same* org (e.g. `*.lightning.force.com` ↔ `*.my.salesforce.com`) does **not** re-bind, and switching to a **non-Salesforce** tab leaves the panel on its last org rather than blanking it. The org-follow decision is a pure, unit-tested helper (`lib/sf-panel.ts` `shouldRebindPanel`); the background service worker owns the `chrome.tabs` events and broadcasts the active tab to the panel, which re-binds. The panel is also **auto-enabled only on Salesforce tabs** (`chrome.sidePanel.setOptions({ tabId, enabled })`, gated on the same allow-listed SF hosts) so it's offered where it's useful and disabled on unrelated tabs. A new **Default tool surface** setting (Options › Appearance; `settings.defaultSurface`, default `modal` so nothing changes unless opted in) lets you prefer the docked panel over the classic centered modal — honoured in this PR by the toolbar popup, which promotes **"Open side panel"** to the primary action when set to `panel`. All of this is Chrome-only and guarded on `chrome.sidePanel`/`chrome.tabs` so the Firefox `sidebar_action` build is unaffected. **No new permission** (the `sidePanel` permission shipped in PR-1; the tab-follow and per-tab enablement ride on existing `host_permissions` — no `tabs` permission added), no new data access, and no new runtime dependency. Deep routing of an on-page tool straight into the open panel (present-view across the content-script↔panel boundary) is a deliberate follow-up.
- **Docked side panel — Workspace-in-a-dock (P2-3, PR-1 of 2).** A new `chrome.sidePanel` host (`entrypoints/sidepanel`) runs the full Workspace — the same tool set, tabbed pane host, org switcher, and present-view sink — inside a docked side panel beside the Salesforce page, so your tools stay open next to the tab you're working on. Open it from the toolbar popup's new **"Open side panel"** button (a live user gesture calls `chrome.sidePanel.open({ tabId })`); the toolbar button still opens the action popup as before (no `openPanelOnActionClick` hijack). Org binding is **bind-on-open**: the panel binds to the org of the tab it was opened from (the active tab's URL), falling back to your last-used org, then an org picker — identical to the Workspace. The shared boot/layout/tool-registration is factored into a chrome-free `ui/workspace-host.ts` that both the Workspace tab and the panel import, so the two can't drift and the Workspace behaves exactly as before. Firefox parity ships too: a per-browser manifest emits `sidebar_action` (native sidebar, no extra permission) pointing at the same page. Adds exactly one new permission, **`sidePanel`** (Chrome only; ledgered, `PRIVACY.md` + store listing updated) — no new data access and no new host permission. Org-follow-across-tab-switches, auto-enable-on-Lightning, and routing the ⚡ menu into the panel are PR-2 (browser-validated). No new runtime dependency.
- **Schema Browser — SOQL insert + export-for-prompt field selection (P2-1, PR-3 of 3).** The Schema Browser field table now integrates with two sibling tools. Each field row gains an **"Insert into query"** quick action (next to "Copy API name") that drops the field's API name into the live SOQL Runner draft — appended into the SELECT list when the runner is open, or stashed as a pending fragment and applied on the next open (mirrors the existing Saved-SOQL pending-query hand-off). The runner exposes a small imperative `insertFieldIntoDraft(field)` hook, threaded into the Schema Browser from both `content.ts` and the Workspace factory; when the hook is absent the action is hidden. For LLM prompts, the field table now **pulls back every field by API name, all pre-selected** (a labelled, keyboard-reachable checkbox per row), and you **unselect** the ones you don't want, with **Select all** / **Clear all** affordances and a live "N of M selected" count. An **"Export selected for prompt"** action emits only the chosen fields through the existing `export-for-prompt` tool — `buildSchemaMarkdown` now takes an optional field subset and falls back to the full-object export when none is supplied (the existing one-shot export path is unchanged). A per-object selection store (object API name → set of field API names) preserves each object's choices across navigation. No new runtime dependency, no new permission.
- **Command palette — custom shortcuts (P2-2, PR-3 of 3).** You can now define your own **name → URL shortcuts** that appear in the ⚡ command palette under the **Shortcuts** category. A new **Custom shortcuts** section on the Options page manages them: add rows, edit a name/URL inline, and remove — saved through the existing single Save button. Validation rejects a **duplicate name** and a **malformed URL** (the same `z.string().url()` boundary the setting is stored behind), so a bad entry is caught on save rather than opened. Shortcuts persist as a top-level `customShortcuts` preference in `chrome.storage.local` (a global preference like theme/bridge, not a per-feature toggle). Selecting a shortcut opens its URL in a **new tab** via `window.open` (never navigating away from the Salesforce page, never `eval`), and — because it flows through the same recent-first candidate pool — a used shortcut resurfaces in the palette's **Recent** section on reopen. No new runtime dependency, no new permission.
- **Command palette overlay + wiring (P2-2, PR-2 of 3).** The ⚡ command palette is now a real, keyboard-first overlay (`ui/command-palette.ts`) that mounts into the shared closed shadow root. A labelled search box drives a `role="listbox"` of `role="option"` rows grouped under per-category headers (Recent / This record / Features / Setup / Shortcuts / Objects); candidates and their recent-first ordering come from PR-1's `lib/palette-sources.ts` and ranking from `lib/fuzzy.ts` — nothing is re-implemented. Full keyboard contract: open focuses the search box, ArrowUp/Down move the active option via `aria-activedescendant`, Enter runs it, Esc closes and restores focus to wherever it was, Tab is trapped inside, and a click outside dismisses (shadow-safe via `composedPath()`). Opening does **zero** Salesforce API calls — features/setup/shortcuts/recents are synchronous, and the Objects category kicks off the shared describe cache only *after* first paint, filling in when it resolves (AC-1). The `open-palette` keyboard command (`Ctrl/Cmd+Shift+K`) and a new persistent **"View all features"** entry in the ⚡ menu both open it; the ⚡ menu stays the dynamic location-aware shortlist while the palette is the exhaustive searchable view (both surfaces kept). Executing a feature dispatches it, a setup/shortcut navigates, a record opens the inspector, and an object opens the schema browser when that feature is present, else the Object Manager page. The palette is itself a kill-switchable registry feature (`command-palette`, default on); when it's disabled the keyboard command falls back to the ⚡ menu. No new runtime dependency, no new permission.
- **Command palette foundations (P2-2, PR-1 of 3).** Groundwork for the ⚡ command palette, with no user-visible change yet (no overlay, no new keyboard redirect). A hand-rolled, dependency-free fuzzy matcher (`lib/fuzzy.ts`) ranks candidates across four tiers (exact/prefix > substring > word-boundary acronym > loose subsequence), scores an api-name key and a label key taking the better of the two, and exposes a stable descending sort (equal scores keep input order). The Setup deep-link map (the `setup-tabs` tab definitions, incl. "Login as user…") is extracted into `lib/setup-links.ts` as one shared source of truth — `setup-tabs` now imports it with zero behaviour change. A pure provider layer (`lib/palette-sources.ts`) assembles categorized palette candidates (features, setup links, custom shortcuts, and a record-Id inspect entry) from injected inputs, applying recent-first ordering; the Objects category is a deliberate empty stub until the describe-cache lands. The enabled-for-context feature filter (available ∩ registered ∩ not kill-switched ∩ user-enabled) is factored out of the side-menu provider into a shared `enabledFeatureIds()` helper so the menu and palette can't drift (AC-3). A small MRU store (`lib/palette-recents.ts`) persists recent selections in `chrome.storage.local`. No new runtime dependency, no new permission.
- **Schema Browser tool (P2-1, PR-2 of 3).** A new two-pane **Schema Browser** — a searchable, windowed object list on the left and a per-object field table on the right — available as a Workspace tool and from a record page via the ⚡ menu. The object list is hand-rolled windowed (renders a bounded page of rows and extends on scroll or when a filter narrows the set), so an 800+ object org never builds an unbounded DOM; no virtualization library. The field table shows label, API name, type, length, required, inline-expandable picklist values, formula source, flattened compound components, and reference targets rendered as clickable cross-links that navigate the tool to the target object; the selected object's child relationships are listed below. Each field has a "Copy API name" quick action (`navigator.clipboard`). An imperative `openFor(sobject)` seeds the record-page entry and powers the reference/child cross-links. All describes flow through the shared session describe cache (`lib/describe-cache.ts`) and the pure viewmodel mappers (`lib/schema-viewmodel.ts`) — no second describe path — so the `sid` stays in the worker. Esc closes, focus is trapped (modal mode) and restored on close, controls are labelled, and the whole tool is keyboard-reachable. No new runtime dependency, no new permission.
- **Shared describe cache + schema viewmodel core (P2-1, groundwork).** The metadata describe cache that powers SOQL Runner autocomplete is now a **session-scoped singleton** hoisted into `lib/describe-cache.ts`, keyed by org origin and reused across every caller — so an sObject described once (globals or a specific object) is not re-fetched by the next consumer in the same session (satisfies P2-1 AC-5). The describe interfaces are extended additively (field `length`/`precision`/`scale`/`calculatedFormula`/`custom`, plus top-level `childRelationships`), and a new pure `lib/schema-viewmodel.ts` maps a describe into filterable object-list and field-table viewmodels (compound-field flattening, picklist expansion, reference-target resolution, formula-source extraction, child-relationship list) with zero DOM. No UI yet, no new permission, no network egress — describes still run through the worker-proxied API client, so the `sid` never leaves the worker.
- **SOQL Runner — "Export all as CSV" (P1-4).** Alongside the existing "Export CSV" (visible page only), the runner now has an **Export all as CSV** action that re-runs the current query and follows Salesforce pagination (`queryMore`/`nextRecordsUrl`) to the end, downloading every row in one file. A live progress indicator (`aria-live` status, "Exporting all… N rows across M pages") and a keyboard-reachable **Cancel** button let you watch and abort a long export; cancelling stops further paging and discards the result so no file is downloaded — an already in-flight page fetch completes first (it runs in the background worker and can't be network-aborted through the proxy), then the export halts before the next page and produces no download. Memory-safe by construction: each fetched page is converted to a CSV text chunk and pushed onto a `parts[]` array — the raw record objects are discarded per page and the file is assembled as a `Blob` from those chunks, so nothing accumulates the whole result set as one giant in-memory string. Runs entirely org-side through the worker-proxied API client; the session id never leaves the worker.
- **DebugLevel picker on Execute Anonymous Apex (P1-5).** When log capture is on, the Execute Anonymous panel now shows a **Log level** `<select>` so you choose which org DebugLevel the captured log uses without a trip to Setup. Options are the org's own DebugLevels (loaded via the worker-proxied Tooling query — the `sid` stays in the worker), plus the default **"SFDT Finest (auto)"** which keeps the previous behaviour (the feature-managed `SFDT_Finest` DebugLevel, created on demand). The chosen level is applied to your DEVELOPER_LOG trace flag before the run — a pick that differs from the currently active flag updates it in place — so the produced log reflects your selection. The selection persists per user in `chrome.storage.local`. The control is a native labelled `<select>` (full keyboard path, both themes via `var(--sfdt-color-*)` tokens).
- **SOQL Query Runner — "Explain" (query plan).** The SOQL Runner now has a **🔎 Explain** action next to Run that fetches Salesforce's query plan for the current query without executing it — the REST `?explain=` query-plan endpoint in REST mode, and the Tooling query-plan endpoint in Tooling mode, both through the worker-proxied API client (the `sid` never leaves the worker). The plan renders as a small table per candidate access path (the first is the one the optimizer chose): **cardinality**, **SObject cardinality**, **leading operation**, **relative cost**, and optimizer **notes**. A non-explainable query (e.g. SOSL) surfaces its error inline in the runner's `role="alert"` panel rather than as a thrown toast.
- **SOQL query template pack (P1-2).** The Saved SOQL panel now shows a read-only **"Templates"** group — a built-in library of useful admin/dev queries, visually distinct from your own bookmarks (brand-accent border + a "Built-in" tag) and clicking Load copies the query straight into the SOQL Runner. Ships 8 templates, each valid SOQL that runs on a vanilla Developer Edition org against standard objects: Apex test coverage, validation rules, login history, setup audit trail, API/usage-limit snapshot (`TenantUsageEntitlement`), recent deploy requests, active TraceFlags, and a record-type list. Templates are pure data (`features/soql-templates.ts`, one array) with no delete affordance, so built-ins can't be removed; a unit test asserts every template is structurally valid SOQL. No new permission, no network egress — queries still run through the existing worker `api` proxy so the `sid` never leaves the worker.
- **"Login as user…" setup tab (P1-9).** The `setup-tabs` feature now exposes a **Login as user…** entry that deep-links to Setup's user list (`/lightning/setup/ManageUsers/home`) — the page where Salesforce renders the per-user "Login" action. It is a deep link only: the extension never lists users or mints sessions itself, and no session id is touched (a plain in-page navigation). Salesforce enforces the Login-As permission server-side. Respects the existing setup-tabs kill-switch and grouping toggle; no new manifest permission.
- **Debug Logs — auto-refresh + bulk delete (P1-6).** The Debug Logs viewer gains an **Auto-refresh (15s)** toggle in its toolbar — off by default; when on, the log list re-queries every 15 seconds so new logs appear while a trace flag is active. The poll interval is owned at feature scope and cleared on view close and on feature teardown (kill-switch / route change), so no orphan timer survives. A **Delete all logs** button clears **every** ApexLog row in the org via the Tooling API behind a count-confirm dialog ("Delete N logs?") — the count is the true org-wide total, fetched by paginating `SELECT Id FROM ApexLog` past the 2000-row first page, and the deletes are issued in chunks so a large org doesn't hang the UI. The accessible dialog carries `role="dialog"`/`aria-modal`, closes on Esc (capture phase) and backdrop click, traps Tab between its two buttons, and restores focus to the opener; the list refreshes after. The `sid` stays in the worker throughout (existing proxied `api` client).
- **Raw-JSON toggle on Inspect Record (P1-7).** The Inspect Record modal now has a **Fields / JSON** view toggle. The JSON view renders the raw REST record payload — the exact object returned by the sObject row fetch, a superset of the editable fields grid — pretty-printed in a `<pre>` (via `textContent`, no `innerHTML`) with a one-click **Copy JSON** button. The toggle is keyboard-reachable (native `<button>`s exposing `aria-pressed`), and the chosen view is preserved while the modal stays open. Inspector-parity; no new permission, and the payload was already fetched through the worker proxy, so the `sid` is untouched.
- **Session-layer hardening — GovCloud, Defender proxy, and China org support (P0-5).** The background worker's Salesforce API resolution (still entirely worker-side; the `sid` never leaves the worker) now handles orgs served on non-standard domains and resolves the true instance more robustly:
  - **New host coverage** — US Government Cloud/GovCloud (`.mil`: `<org>.my.salesforce.mil` / `.lightning.force.mil`), Salesforce China (`.sfcrmapps.cn`), and orgs fronted by a **Microsoft Defender for Cloud Apps** reverse proxy (`.mcas.ms`). Ledgered new manifest `host_permissions` — `https://*.my.salesforce.mil/*`, `https://*.lightning.force.mil/*`, `https://*.sfcrmapps.cn/*`, `https://*.mcas.ms/*` — and matching cookie-read allowlist entries (least-privilege; only the `sid` cookie of these hosts is read, locally). Defender-proxied sessions correctly authenticate back through the proxy origin.
  - **SID-prefix (Org ID) cross-matching.** When several candidate base domains are in play, the worker cross-matches each candidate's `sid` prefix (the 15-char Org ID) against the page origin's, dropping any candidate whose cookie belongs to a *different* org you're also logged into — so a request is never authenticated with the wrong org's session. Fixes exotic domains where the naive host guess was wrong.
  - **Per-host session cache** in `chrome.storage.session` (memory-only, cleared when the browser closes; never `chrome.storage.local`, and never the `sid` — only the resolved API hostname + org id). Repeat calls read a single cookie instead of re-scanning candidate domains; a 401 or org switch invalidates the entry and re-resolves with no user action.
  - **Incognito isolation** — the manifest now declares `incognito: "split"`, so an incognito window resolves and caches its own sessions instead of sharing the normal profile's.
- **Dark mode.** The extension UI now has a dark theme, cashing in the P0-1 token consolidation — dark values for every `--sfdt-color-*` token live alongside the light ones in `lib/tokens.ts`, emitted under a `:root[data-sfdt-theme="dark"]` block plus a `@media (prefers-color-scheme: dark)` fallback. A new **Appearance › Theme** setting on the options page chooses `light`, `dark`, or `auto` (default); `auto` follows your OS live via a `prefers-color-scheme` listener, while a manual `light`/`dark` overrides the OS. The choice is a single global preference in `chrome.storage` (`settings.theme`), read on boot and applied on all three surfaces — the injected content-script UI (side menu, tools, modals), the Workspace app, and the options page — and it persists across browser restarts. Theming is scoped to the extension's own `--sfdt-*`-styled overlays; the host Salesforce page is untouched. Body text meets WCAG AA contrast in both themes. P0-1 had overloaded a few tokens (used as both a saturated fill and as coloured text); dark mode splits the foreground role onto dedicated tokens (`color-on-accent`, `color-text-strong`, `color-brand-text`, `color-error-text`, `color-success-text`) whose light values are identical to what they replaced, so the light theme is visually unchanged.
- **UX & accessibility conventions doc** (`extension/CONVENTIONS.md`) — a numbered a11y/overlay/keyboard/theme/shadow-root checklist that reviewer agents apply verbatim to every UI-bearing change, plus two annotated reference implementations (`setup-tabs`, `flow-list-search`). Codifies the overlay rules (Esc closes, focus trap + restore, dialog semantics, pin/click-outside dismiss), the `createElement`+`textContent`/zero-`innerHTML` DOM discipline, and the `var(--sfdt-color-*)` theme-token rule. Contributor-facing; ships no runtime code.
- **Packaging version-match guard** (`scripts/check-version-match.ts`) — the `package:ext` step now fails if `extension/package.json`'s version differs from the built manifest (`.output/chrome-mv3/manifest.json`), naming both versions and the file, so a stale build artifact can't be zipped and shipped. Runs in the Extension CI packaging job; documented in the new `extension/RELEASING.md`.
- **Toolbar action popup + keyboard shortcuts.** Clicking the extension's toolbar button now opens a thin popup showing, for the active tab: whether it's a Salesforce page and which org, a live-session status dot, a bridge (sfdt CLI) status dot, quick buttons for the Workspace / quick menu / Settings, and the extension version. On a non-Salesforce tab it shows a "not a Salesforce tab" state and makes zero org/API calls. Three declared keyboard commands are also added and are remappable at `chrome://extensions/shortcuts`: **Open Workspace** (`Ctrl/Cmd+Shift+E`), **Open quick menu** (`Ctrl/Cmd+Shift+K`, opens the ⚡ side menu until the command palette ships), and **Toggle inspector** (declared now; surfaces a "not available yet" notice until the LWC inspector lands). Because the commands are registered by the browser rather than by in-page keydown listeners, they work on a Salesforce tab even before the content script has finished loading. Session status is read in the service worker, never from the page, so the session cookie stays in the worker. No new permission is required — `commands` is a manifest key, not a permission.
- **Right-click "Inspect this record" context menu (P1-8).** Right-clicking a Salesforce page — or a link to a Salesforce record — now offers **"SFDT: Inspect this record"**, which opens the record inspector for the record Id found in the page URL or the right-clicked link's target. The item appears only on your Salesforce hosts (scoped via `documentUrlPatterns`/`targetUrlPatterns`) and does nothing when no 15/18-char record Id is present. It's a normal kill-switchable feature with an opt-in **Options → Features** toggle (enabled by default). The menu lives in the service worker and only ever forwards a record Id to the inspector — it reads no cookies or session data, so the `sid` stays in the worker. **Ledgered new manifest permission `contextMenus`** (documented in `PRIVACY.md` and the store listing); no other permission added.
- **SOQL Runner — Copy JSON + Copy for Excel.** The result toolbar gains two client-side copy actions alongside Copy/Export CSV: **Copy JSON** copies the currently displayed rows as a pretty-printed (2-space) JSON array (the Salesforce `attributes` envelope stripped so it matches the shown columns; nested relationship records keep their structure), and **Copy for Excel** copies them as tab-separated values with a header row that paste straight into a spreadsheet as columns. TSV values containing tabs, newlines, or quotes are quoted/escaped so a cell can't break the grid. Both operate purely on the already-fetched result set (the loaded pages, up to the pagination cap) with no extra org calls, and use `navigator.clipboard` (no new permission).

### Changed
- **Security posture: the Salesforce session id (`sid`) no longer enters page-adjacent memory.** Every REST/Tooling/SOAP call now runs inside the background service worker via a new `sfApiFetch` message route (`lib/sf-api-proxy.ts`): the worker reads the `sid` cookie, injects `Authorization`, performs the dual-host fallback + one-time 401 retry, and returns only the response *text* — never the sid. `lib/salesforce-api.ts` is now a thin client that describes the call and parses the reply; it holds no sid and makes no Salesforce fetch of its own. SOAP calls send a sentinel placeholder that the worker swaps for the real sid inside the `SessionHeader` only. Request headers are allowlisted (any client-supplied `Authorization` is stripped). A guard test (`test/sid-never-leaves-worker.test.ts`) plus a scoped ESLint rule enforce that feature/UI code never touches `chrome.cookies`, the `getSidForUrls` route, `getSessionDetails`, or a raw `Authorization: Bearer` header. Internal refactor — no user-facing behaviour change.
- **SID-in-worker migration complete — zero exceptions.** The Event Streaming Monitor's CometD/Bayeux long-poll (the last page-side holder of a raw `sid`) now runs entirely in the service worker behind a long-lived `sfApiStream` Port (`lib/sf-stream-worker.ts`). The feature opens the Port, sends `subscribe`/`unsubscribe`, and renders only the `status`/`event` messages the worker pushes back — the sid never reaches the page. The residual page-side session machinery (`getSession`/`getSessionDetails`/`getSidForUrls` client path) is deleted, and both the guard test and the ESLint rule now allowlist nothing (they also fail on any `getSessionDetails` reference). This completes the P0-4 "SID never leaves the worker" guarantee. On MV3 service-worker eviction the Port disconnects and the monitor shows "Disconnected" with Subscribe re-enabled, so the user can reconnect. No new manifest permission (`chrome.runtime.connect` is a platform primitive). Internal refactor — no user-facing behaviour change. (Re-landed onto `develop` after the original approved change was stranded by a squash-merge into an intermediate branch that never reached `develop`.)
- **Shadow-DOM host for injected UI (internal, no visual change).** All content-script-injected UI on a Salesforce page — the ⚡ side button and its menu, `present-view` page-mode modals, and toasts — now renders inside ONE shared **closed** shadow root (`ui/shadow-host.ts` + `ui/content-root.ts`), fully isolating it from the host page's CSS in both directions: Salesforce's global styles can't bleed into our overlays, and our styles can't leak onto the page. This cuts the per-release visual breakage caused by Salesforce DOM/CSS churn. Dark mode is unaffected — the `--sfdt-*` design tokens stay defined on the host `:root` (custom properties inherit across the shadow boundary), so `var(--sfdt-*)` still resolves and flipping `data-sfdt-theme` on the host still re-themes the overlays live. Click-outside dismissal now uses `composedPath()` so it works correctly across the shadow boundary; Esc/focus behaviour is unchanged. Our own full-page surfaces (the Workspace app + options page) are not injected into a hostile document and stay in light DOM. Realises CONVENTIONS.md item 13 (previously a documented gap).
- **Single Salesforce API-version constant.** Divergent `v62.0` literals across features are consolidated into one authoritative `SF_API_VERSION` (`lib/api-version.ts`), re-exported by the client as `api.apiVersion`.
- **Design-token refactor (internal, no visual change).** Every hard-coded colour across the content-script features, the Workspace app page, the options page, and the shared `ui/` components (~770 hex literals in 38 files) is now a `var(--sfdt-color-*)` CSS custom property. The raw values live in one place — `lib/tokens.ts` — which each surface injects at boot (`ensureTokens()` on Salesforce pages; the `:root` token block prepended to the app/options stylesheets). This is a pure refactor with no user-facing change; it's the enabling step for a future single-switch dark theme, which can now supply dark values in one file instead of hunting colours across the codebase. One documented exception remains: the user-configurable `canvasSearch.highlightColour` default (`#FFD700`) stays a literal because it's runtime data string-concatenated with an alpha suffix and cannot be a CSS variable.
